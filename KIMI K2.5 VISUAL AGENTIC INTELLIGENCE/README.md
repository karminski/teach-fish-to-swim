
# KIMI K2.5: VISUAL AGENTIC INTELLIGENCE

## 革新

Kimi K2.5 是 Moonshot AI 开源的首个多模态智能体模型，标志着**通用智能体(General Agentic Intelligence)**的重大突破。与以往模型不同，K2.5 实现了**文本与视觉的联合优化**，让两种模态相互增强而非相互妥协。更令人惊艳的是其创新性的 **Agent Swarm（智能体群）**框架——通过并行编排多个专业化子智能体，将复杂任务分解为异构子问题并发执行，**在广域搜索场景中将推理延迟降低高达 4.5 倍**，同时将 Item-F1 从 72.8% 提升至 79.0%。这一突破彻底改变了传统智能体"串行执行"的瓶颈，为超大规模、高复杂度的智能体工作负载开辟了新范式。

## 论文解析

### 核心创新：文本与视觉的联合优化

Kimi K2.5 的第一大核心是**打破了传统多模态模型的"后融合"范式**。以往的视觉-语言模型通常在文本 LLM 训练后期才引入视觉 Token（占比往往高达 50% 以上），这种做法本质上是把多模态能力当作"附加功能"。

#### 颠覆性发现：早期融合 + 低视觉比例更优

K2.5 团队通过大规模消融实验发现了一个**反直觉的结论**：

| 视觉注入时机 | 视觉:文本比例 | 视觉知识 | 视觉推理 | OCR | 文本知识 | 文本推理 | 代码 |
|------------|------------|---------|---------|-----|---------|---------|-----|
| 早期 (0%) | 10%:90% | **25.8** | **43.8** | **65.7** | **45.5** | **58.5** | **24.8** |
| 中期 (50%) | 20%:80% | 25.0 | 40.7 | 64.1 | 43.9 | 58.6 | 24.0 |
| 后期 (80%) | 50%:50% | 24.2 | 39.0 | 61.5 | 43.1 | 57.8 | 24.0 |

**核心洞察**：
- 在固定总 Token 预算下，视觉比例对最终性能影响极小
- **早期以较低比例融合视觉，效果反而最佳**
- 传统"后期大比例"策略会导致"先降后升"现象——引入视觉时文本能力暂时下降

**原因**：早期融合让模型从一开始就学习统一的多模态表示空间，避免了后期"模态域迁移"造成的表示崩溃。

### Zero-Vision SFT：文本激活视觉能力

这是 K2.5 最违反直觉的创新之一：**在监督微调阶段，只用文本数据就能激活视觉推理和工具使用能力**。

#### 设计哲学

预训练的多模态模型虽然具备视觉理解能力，但不会主动进行视觉工具调用。传统做法是使用人工标注或提示工程生成的视觉 CoT 数据，但这种数据：
- 多样性受限
- 往往局限于简单图表
- 工具操作原始（仅有 crop、rotate、flip 等）

#### 解决方案

K2.5 的做法是：**所有图像操作都通过 IPython 中的程序化操作代理**。这实际上是传统视觉工具使用的泛化：
- 通过二值化和计数进行像素级操作（如物体尺寸估计）
- 泛化到视觉定位、计数、OCR 等任务
- 只需文本 SFT 即可激活这些能力

![零视觉 SFT 后的视觉 RL 训练曲线](./assets/images/Vision%20RL%20training%20curves%20on%20vision%20benchmarks.png)

**为什么可行？** 因为预训练阶段已经建立了强大的视觉-文本对齐，能力可以自然地跨模态泛化。实验表明，零视觉 SFT 的泛化性能**优于**传统的文本-视觉联合 SFT。

### 联合多模态强化学习：视觉增强文本

K2.5 在后训练阶段采用了联合 RL 策略，更令人震惊的是：**视觉 RL 不仅没有损害文本性能，反而显著提升了它**。

#### 基于结果的视觉 RL

在 Zero-Vision SFT 之后，模型仍存在缺陷：
- 有时忽略视觉输入
- 该看图时不看图

解决方案：在显式需要视觉理解的任务上进行基于结果的 RL，包括：
- **视觉定位与计数**：对象的精确定位和枚举
- **图表与文档理解**：结构化视觉信息解释和文本提取
- **视觉关键型 STEM 问题**：筛选出必须依赖视觉输入的数学和科学题目

#### 惊人发现：视觉 RL 提升文本能力

| 基准 | 视觉 RL 前 | 视觉 RL 后 | 提升 |
|------|-----------|-----------|-----|
| MMLU-Pro | 84.7 | **86.4** | +1.7 |
| GPQA-Diamond | 84.3 | **86.4** | +2.1 |
| LongBench v2 | 56.7 | **58.9** | +2.2 |

**原因分析**：视觉 RL 增强了模型在结构化信息提取方面的校准能力，降低了对类似视觉定位推理的查询的不确定性（如计数、OCR）。这表明**视觉 RL 贡献了跨模态泛化能力，改善文本推理而不会退化语言能力**。

#### 联合多模态 RL 范式

受上述发现启发，K2.5 采用了全新的 RL 组织方式：
- **不按输入模态划分**专家（传统做法）
- **按能力划分**：知识、推理、编码、智能体等
- 每个领域专家同时学习纯文本和多模态查询
- 生成奖励模型(GRM)也跨异构轨迹优化，无模态壁垒

**效果**：通过任一模态获得的能力改进，天然地泛化到另一模态的相关能力，最大化跨模态能力迁移。

### Agent Swarm：并行智能体编排

传统智能体系统的致命缺陷是**串行执行**推理和工具调用步骤。即使是能进行数百步推理的系统（如 Kimi K2-Thinking），也受困于线性时间扩展，导致：
- 不可接受的延迟
- 任务复杂度受限

当智能体工作负载扩展到大规模研究、设计、开发等场景时，串行范式效率极低。

#### Agent Swarm 架构

![Agent Swarm 架构图](./assets/images/An%20agent%20swarm%20has%20a%20trainable%20orchestrator%20that%20dynamically%20creates%20specialized%20frozen%20subagents%20and.png)

**核心设计**：
- **可训练的编排器(Orchestrator)**：负责任务分解和调度
- **冻结的子智能体(Frozen Subagents)**：从固定的中间策略检查点实例化
- **动态任务分解**：将复杂任务拆解为异构子问题
- **并行子任务调度**：子智能体并发执行

**关键理念**：并行性**不是默认有利的**——是否、何时、如何并行化，都是通过环境反馈和 RL 驱动的探索**显式学习**的。

#### Parallel-Agent RL (PARL)

![并行智能体 RL 训练进展](./assets/images/In%20our%20parallel-agent%20reinforcement%20learning%20environment,%20the%20training%20accuracy%20increases%20smoothly%20as%20training%20progresses.png)

**为何解耦架构？**

避免端到端协同优化的两大挑战：
1. **信用分配模糊性**：正确答案不意味着所有子智能体执行完美，失败也不代表所有子智能体出错
2. **训练不稳定性**：多智能体联合优化的梯度极不稳定

**解决方案**：
- 冻结子智能体，将其输出视为**环境观测**而非可微决策点
- 解耦高层协调逻辑与低层执行熟练度
- 先用小型子智能体训练编排器，再过渡到大型模型

#### PARL 奖励函数

```
r_PARL(x, y) = λ₁ · r_parallel + λ₂ · r_finish + r_perf(x, y)
              ⎣___________⎦   ⎣__________⎦   ⎣__________⎦
              实例化奖励      子智能体完成率    任务级结果
```

**各项作用**：
- **r_parallel**：鼓励子智能体实例化，缓解"串行坍缩"（编排器默认单智能体执行的局部最优）
- **r_finish**：奖励成功完成的子任务，防止"伪并行"（大量创建无意义子智能体刷并行指标）
- **r_perf**：评估整体解决方案的成功度和质量

训练过程中，λ₁ 和 λ₂ 逐渐退火至零，确保最终策略优化主要目标。

#### 关键步数(Critical Steps)

类比计算图中的关键路径，定义并行智能体环境中的时间成本：

```
CriticalSteps = Σ[t=1 to T] (S_main^(t) + max_i S_sub,i^(t))
```

- 每个阶段 t，主智能体执行 S_main^(t) 步
- 若并行启动多个子智能体，该阶段持续时间由**最长运行的子智能体**决定
- 总关键步数 = 所有阶段的关键路径之和

**效果**：通过约束关键步数而非总步数，框架显式激励有效并行化。过度创建不减少最大并行分支执行时间的子任务，收益极小；而良好平衡的任务分解可直接缩短关键步数。

#### 并行能力诱导提示构建

构建一套合成提示，强调串行智能体执行的极限：
- **广域搜索**：需同时探索大量独立信息源
- **深度搜索**：需多个推理分支并延迟聚合
- **真实工作负载启发**：长上下文文档分析、大规模文件下载

**设计理念**：这些任务串行执行时难以在固定推理步数和工具调用预算内完成。提示本身**不显式指示并行化**，而是塑造任务分布，使并行分解和调度策略自然受到青睐。

### 架构与训练细节

#### MoonViT-3D：统一图像与视频

**核心创新**：将 NaViT 的"patch n' pack"理念扩展到时间维度。

**工作原理**：
- 最多 4 个连续帧视为时空体
- 2D patch 从这些帧中联合展平并打包成单个 1D 序列
- 相同的注意力机制无缝地跨空间和时间运作
- 轻量级时间池化在 MLP 投影器前进行，产生 **4× 时间压缩**

**优势**：
- 完全共享参数和嵌入空间
- 从静态图像到动态视频的知识泛化最大化
- 无需专门的视频模块或架构分支
- 可处理长达 4 倍的视频（在相同上下文窗口内）

#### Token 高效强化学习：Toggle 算法

**问题**：测试时计算扩展的核心是在计算与推理质量间权衡。但强化学习中的长度过拟合现象：严格预算约束下训练的模型无法泛化到更高计算规模，默认截断推理模式。

**解决方案**：Toggle 启发式算法，每 m 次迭代交替两个优化阶段：

**Phase 0（预算限制阶段）**：
```
r̃(x, y) = r(x, y) · I[mean_accuracy(x) < λ OR |y| ≤ budget(x)]
```
- 训练模型在任务相关 Token 预算内解决问题
- 仅当平均准确率超过阈值 λ 时强制执行（防止过早牺牲质量换效率）

**Phase 1（标准扩展阶段）**：
```
r̃(x, y) = r(x, y)
```
- 模型生成响应直至最大 Token 限制
- 鼓励模型利用计算实现更好的推理时扩展

![Token 高效 RL 效果](./assets/images/Comparison%20of%20model%20performance%20and%20token%20usage%20for%20Kimi%20K2%20Thinking%20following%20token-efficient%20RL.png)

**效果**：
- 在所有基准上输出长度平均减少 25~30%
- 性能影响可忽略不计
- 减少 CoT 中的冗余模式（重复验证、机械计算）
- 强域泛化：仅在数学和编程上训练，在 GPQA 和 MMLU-Pro 上也实现一致的 Token 减少

#### Decoupled Encoder Process (DEP)

**问题**：传统多模态训练中，视觉编码器和文本嵌入共处于流水线并行(PP)的第 0 阶段。由于多模态输入大小变化（图像数量、分辨率），Stage-0 面临：
- 计算负载剧烈波动
- 内存使用大幅变化

**解决方案**：利用视觉编码器的独特拓扑位置（前向传播起点，反向传播终点），采用 DEP 三阶段：

1. **平衡视觉前向**：
   - 在所有 GPU 上复制小型视觉编码器
   - 根据负载指标（图像/patch 计数）在所有 GPU 间均匀分配前向计算
   - 丢弃所有中间激活，仅保留最终输出激活（降低峰值内存）
   - 结果收集回 PP Stage-0

2. **主干训练**：
   - 对主 Transformer 主干执行前向和反向传播
   - 完全利用纯文本训练中验证的高效并行策略

3. **视觉重计算与反向**：
   - 重新计算视觉编码器前向传播
   - 执行反向传播以计算视觉编码器参数的梯度

**效果**：
- 实现负载平衡
- 解耦视觉编码器与主干的优化策略
- K2.5 无缝继承 K2 的并行策略
- **多模态训练效率达到纯文本训练的 90%**

#### 动态子智能体创建与调度

通过 PARL，编排器学习自适应策略：
- **动态实例化**而非预定义子智能体
- 根据查询推理所需的数量、时机和专业化
- 有机地涌现出异构智能体组（见论文 Figure 6）

**与静态分解的区别**：学习策略使编排器能够基于任务结构和问题状态进行推理，创建响应性强的智能体编排。

#### Agent Swarm = 主动上下文管理

Agent Swarm 本质上是**多智能体架构赋能的主动式智能上下文管理**，区别于传统反应式策略：

**传统策略**（反应式）：
- Hide-Tool-Result
- Summary
- Discard-all

这些方法在上下文溢出时通过压缩或丢弃累积历史来应对，虽能减少 Token 使用，但：
- 固有的反应性
- 经常牺牲结构信息或中间推理

**Agent Swarm**（主动式）：
- 通过显式编排实现主动上下文控制
- 长期任务分解为并行、语义隔离的子任务
- 每个子智能体有界的局部上下文
- 子智能体维护独立工作记忆，执行局部推理，不直接改变主编排器的全局上下文
- 仅任务相关输出（而非完整交互轨迹）选择性地路由回编排器

**效果**：
- 诱导**上下文分片**而非上下文截断
- 沿额外架构维度扩展有效上下文长度
- 保持模块化、信息局部性和推理完整性
- 在 BrowseComp 上，Agent Swarm 以更少的关键步数实现更高准确率（见论文 Figure 7）

## 技术亮点深度解读

### 1. 为何 Zero-Vision SFT 有效？

这看似违反直觉，但背后有深刻的科学原理：

**传统观点**：需要视觉 CoT 数据来教会模型视觉推理。

**K2.5 的发现**：
1. **预训练的力量**：15T Token 的联合预训练已建立强大的视觉-文本对齐
2. **能力泛化**：工具使用能力可跨模态迁移
3. **IPython 作为通用接口**：所有视觉操作都可程序化表达

**类比**：就像一个精通英语和法语的人（联合预训练），只需用英语例子（文本 SFT）学会编程，就能自然地用法语编程（视觉推理）。

### 2. 视觉 RL 为何能增强文本能力？

这是 K2.5 最令人惊讶的发现之一。传统认知是不同模态的训练会相互干扰，但 K2.5 证明了相反的情况。

**机制分析**：
- **结构化信息提取**：视觉任务（定位、OCR、图表理解）本质上是结构化信息提取
- **校准改进**：这类训练增强了模型的输出校准能力
- **跨模态迁移**：结构化提取能力自然迁移到文本领域（如从长文档提取关键信息）
- **不确定性降低**：减少了对类似任务的查询不确定性

**深层原因**：联合预训练建立的统一表示空间，使得能力改进可以跨模态传播。

### 3. Agent Swarm 的设计精髓

**为何不端到端训练？**

端到端多智能体训练的两大杀手级问题：
1. **信用分配**：100 步推理后的最终奖励，如何分配给每个子智能体的每个动作？
2. **训练不稳定**：多个策略同时更新，梯度方差爆炸

**解耦架构的优雅**：
- 子智能体 = 环境的一部分（冻结）
- 编排器 = 唯一的可训练策略
- 清晰的责任边界：编排器学"分解与调度"，子智能体提供"执行能力"

**渐进式训练**：
- 先用小模型作为子智能体训练编排器（快速迭代）
- 再切换到大模型（获得强大的执行能力）
- 动态调整实例比例（最大化集群资源利用）

### 4. Critical Steps 的巧妙设计

**为何不用总步数？**

总步数奖励"做更多工作"，但并行的意义是"同时做多件事，节省时间"。

**Critical Steps 的理念**：
```
时间成本 = Σ max(各并行分支的耗时)
```

**效果**：
- 创建 10 个并行子智能体但都要等最慢的一个 → 关键步数 = 最慢的步数
- 精心分解任务，平衡各分支负载 → 关键步数大幅减少
- 自然激励编排器学习"有效的"并行化，而非"虚假的"并行化

### 5. Toggle 算法的辩证法

**矛盾**：
- 要效率 → 严格预算约束
- 但会导致 → 长度过拟合，无法扩展

**Toggle 的解决**：
- Phase 0：学习"预算内解决问题"（效率）
- Phase 1：学习"利用更多计算解决更难问题"（可扩展性）
- 交替优化 → 同时获得两种能力

**哲学意义**：这是一种**双目标优化**的随机交替策略，避免了两个目标间的恶性竞争。

## 架构与数据洞察

### 预训练数据的精心设计

#### 文本数据

延续 Kimi K2 的高质量数据处理管道，K2.5 特别加强：

**增强代码智能**：
- 大幅增加仓库级代码（跨文件推理和架构理解）
- Issues、代码审查、提交历史（真实开发模式）
- 从 PDF 和网页语料库检索的代码相关文档

**效果**：
- 增强仓库级理解，应对复杂编码任务
- 改进智能体编码子任务（补丁生成、单元测试编写）
- 提升代码相关知识能力

#### 视觉数据（7 大类别）

1. **Caption 数据**：基础模态对齐，严格限制合成标题（缓解幻觉）

2. **Image-Text Interleaving**：来自书籍、网页、教程，支持多图理解和长上下文学习

3. **OCR 数据**：跨多语言文本、密集布局、多页文档

4. **知识数据**：通过布局解析器处理的学术材料，发展视觉推理

5. **多模态问题解决语料库**：
   - STEM 领域（K-12 到大学）
   - 通过定向检索和网络爬取聚合
   - 用 ICL 自动将原始材料重构为结构化学术问题

6. **图像-代码配对数据**：
   - HTML、React、SVG 等代码格式
   - 配对的渲染截图
   - 使模型对齐抽象结构逻辑与具体视觉几何

7. **智能体与时间理解**：
   - GUI 截图和动作轨迹（桌面、移动、Web）
   - 人工标注演示
   - 来自不同来源的视频数据（小时级视频理解和细粒度时空感知）
   - 定位数据（边界框、点基准）
   - **新的轮廓级分割任务**（像素级感知学习）

所有数据经过严格的过滤、去重和质量控制。

### 训练基础设施

**硬件**：NVIDIA H800 GPU 集群，8×400 Gbps RoCE 跨节点互连

**并行策略**：
- 16 路流水线并行(PP)，虚拟阶段
- 16 路专家并行(EP)
- ZeRO-1 数据并行
- 可在 32 的任意倍数节点上训练

**内存优化**：
- 选择性重计算（LayerNorm、SwiGLU、MLA 上投影）
- 将不敏感激活压缩为 FP8-E4M3
- 剩余激活 Offload 到 CPU，重叠流式传输

**数据加载基础设施**：
- S3 兼容对象存储
- 保留原生格式的视觉数据
- 高效灵活的加载：
  - **灵活性**：动态打乱、混合、分词、损失掩码、序列打包
  - **增强**：视觉和文本模态的随机增强
  - **确定性**：完全确定性训练，可中断恢复
  - **可扩展性**：分层缓存，扩展至大型分布式集群

### 统一智能体强化学习环境

**Gym-like 接口**：标准化环境实现

**可插拔组件**：
- **Toolset 模块**：支持各种工具和沙盒
- **Judge 模块**：多方面奖励信号
- **提示多样化和指令跟随增强模块**

**执行层**：
- 每个智能体任务 = 独立异步协程
- 递归触发子任务 rollout
- **Rollout Manager**：编排多达 100,000 个并发智能体任务
- 支持部分 rollout 等高级功能

**推理引擎协同设计**：
- 严格遵循 Token-in-Token-out 范式
- 记录所有输出的对数概率（train-inference mismatch 修正）
- 自定义 RL 推理 API
- **LLM Gateway**：代理服务，为黑盒环境记录详细的 rollout 请求和响应

## 对 AGI 的启示

### 1. 多模态不应是"附加功能"

传统做法：
```
文本 LLM (强) → 后期加视觉 (妥协) → 多模态模型 (弱)
```

K2.5 的范式：
```
文本 + 视觉 (从头联合) → 两者互相增强 → 多模态模型 (强)
```

**启示**：真正的多模态 AGI 应该从一开始就以统一方式处理所有模态，而非"拼接"不同模态的专家。

### 2. 串行智能体的天花板

**瓶颈**：
- 任务复杂度 ∝ 串行步数
- 延迟 ∝ 串行步数
- 当步数 → 数百步时，延迟 → 不可接受

**Agent Swarm 的突破**：
- 任务复杂度 ∝ 最长并行分支
- 延迟 ∝ max(各分支)，而非 Σ(各分支)
- 可扩展到极大规模任务

**对 AGI 的意义**：未来的 AGI 系统必须具备**动态编排多个专业化子系统并行工作**的能力，这是处理真实世界复杂任务的必要条件。

### 3. 测试时计算的效率问题

**现状**：测试时扩展 = 更多 Token = 更好性能，但也 = 更高成本

**Toggle 的贡献**：
- 在不牺牲可扩展性的前提下提高效率
- 25~30% Token 减少，性能几乎无损
- 强域泛化能力

**启示**：AGI 不仅要"能够"推理，还要"高效"推理。Token 效率将是测试时计算扩展能否实用化的关键。

### 4. 跨模态能力迁移的潜力

K2.5 证明：
- 文本 SFT → 激活视觉推理
- 视觉 RL → 增强文本能力

**深层含义**：
- 人类智能也是高度整合的，各种能力相互支持
- AGI 应该追求**统一的认知基础**，而非"模态专家的集合"
- 在一个领域的学习应该自然地泛化到其他领域

### 5. 从"单一模型"到"模型编排"

**传统范式**：更大的单一模型 → 更强的能力

**Agent Swarm 的启示**：
- 编排多个专业化模型 > 单一巨型模型
- 关键是**学习如何编排**
- 编排器 + 专业子智能体 = 新的架构范式

**对 AGI 的意义**：未来的 AGI 可能不是"一个超级模型"，而是"一个超级编排器 + 大量专业化模型"的生态系统。

## 局限性与未来方向

### 当前局限

1. **BabyVision 性能**：36.5%，虽超越多数模型，但仍远低于人类（94.1%），表明基础视觉能力仍有提升空间

2. **Agent Swarm 的适用性**：
   - 最适合可并行分解的任务
   - 对于串行依赖性强的任务，收益有限
   - 需要额外的计算资源（多个子智能体并行运行）

3. **子智能体冻结**：
   - 子智能体不可训练，能力固定
   - 无法从编排器的反馈中学习改进

4. **计算成本**：
   - MoE 架构虽激活参数少，但总参数达 1.04T
   - Agent Swarm 虽减少延迟，但总计算量可能更高（多智能体并行）

### 未来研究方向

1. **更强的基础视觉能力**：
   - 参考 BabyVision 基准，加强非语言细节观察、流形理解、空间想象
   - 探索更好的视觉编码器架构

2. **动态子智能体创建**：
   - 当前从固定检查点实例化
   - 未来：运行时动态生成/微调子智能体

3. **层次化 Agent Swarm**：
   - 子智能体也可以是编排器，创建更深的层次
   - 支持更复杂的任务分解

4. **端到端可训练的多智能体**：
   - 解决信用分配和训练稳定性问题
   - 允许子智能体从协作中学习

5. **跨模态能力迁移的理论理解**：
   - 为什么视觉 RL 能增强文本？
   - 如何设计训练过程最大化跨模态迁移？

6. **更高效的并行化策略**：
   - 自动学习最优的任务分解粒度
   - 根据资源约束自适应调整并行度
